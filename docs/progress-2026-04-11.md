## ⚠️ 遇到的问题

### 1. 测试并发冲突

**问题**: 同时运行所有测试时，PostgreSQL 数据库出现并发冲突

**错误示例**:
```
ERROR: duplicate key value violates unique constraint "idx_users_email"
ERROR: relation "users" does not exist
```

**原因**:
- 多个测试进程同时操作同一个数据库
- DROP TABLE + AutoMigrate 存在竞态条件

**解决方案**:
1. 使用不同的数据库名称
2. 添加 `-parallel=1` 禁用并发
3. 为每个测试进程使用独立的数据库 schema

**临时方案**: 分别运行各层测试
```bash
go test ./internal/repository/... -v -cover
go test ./internal/services/... -v -cover
```

---

### 2. Mock 接口 vs 真实数据库

**问题**: Service 层测试使用真实数据库，不是 mock

**选择原因**:
1. **简单快速**: 不需要定义接口和实现 mock
2. **接近真实**: 测试真实的数据交互
3. **Repository 已测试**: Repository 层已经通过测试，Service 层可以依赖

**缺点**:
- 依赖数据库，测试较慢
- 隔离性不如 mock

**长期优化建议**:
1. 考虑为 Service 层定义接口
2. 使用 mockery 等工具生成 mock
3. 提供"快速测试"（mock）和"集成测试"（真实数据库）两种模式

---

### 3. HTTP 客户端 Mock 缺失

**问题**: RSS Service 测试中，`FetchAndSaveArticles` 需要 HTTP 客户端 mock

**当前状态**: 跳过 `TestRSSService_FetchAndSaveArticles_Success`

**解决方案**:
1. 使用 `httptest` 包提供 HTTP mock server
2. 使用 mock HTTP 客户端库（如 `github.com/stretchr/testify/mock`）
3. 使用真实 RSS feed（仅用于集成测试）

**示例代码**（使用 httptest）:
```go
func TestRSSService_FetchAndSaveArticles_Success(t *testing.T) {
```

## 💡 经验教训

### 成功因素

1. **测试分层策略**: 先 Repository，再 Service，从易到难
2. **真实数据库**: 接近真实场景，发现更多问题
3. **测试隔离**: 每个测试前清理数据库表
4. **渐进式覆盖**: 先覆盖核心功能，再逐步补充

### 遇到的挑战

1. **并发测试**: 数据库冲突导致测试失败
2. **Mock 复杂性**: HTTP 客户端 mock 较复杂，暂时跳过
3. **覆盖率目标**: Service 层覆盖率未达预期

### 改进建议

1. **添加 Handler 层测试**: 提升整体覆盖率
2. **Mock HTTP 客户端**: 完善 RSS Service 测试
3. **解决并发问题**: 使用独立 database schema
4. **添加性能基准测试**: 监控测试执行时间
